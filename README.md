# RESTful API приложение для работы с отелями:

## Приложение запускается на порту 8092 из консоли при помощи команды mvn spring-boot:run

## В приложении существует документация Swagger, доступная по пути - /swagger-ui/index.html

## Существует возможность переключения между базами данных: по умолчанию стоит h2, можно поменять профиль и изменить бд на нужную

## Есть тесты. 

## Приложение разделено на 4 независимых слоя:
- API Layer (Controller): HotelController отвечает только за HTTP-протокол, прием JSON и возврат ответов.
- Service Layer (Business Logic): HotelService — сердце приложения. Здесь происходит координация данных, вызов мапперов и транзакционное управление.
- Persistence Layer (Data Access): HotelRepository инкапсулирует работу с БД. Благодаря Spring Data JPA, смена H2 на Postgres происходит здесь.
- Domain/Model Layer: Сущности (Hotel, Address, Amenity) и DTO. Это структуры данных, которые путешествуют между слоями.

## Использование паттернов проектирования:
- Data Transfer Object (DTO): Разделены внутренние сущности БД и объекты, которые отдаются клиенту (HotelShortDto, HotelFullDto). Это защищает API от изменений в БД.
- Data Mapper: Логика преобразования вынесена из Entity в DTO в отдельный класс HotelMapper. Это классический паттерн для уменьшения связности.
- Repository: Использована абстракцию над хранилищем данных. Это позволяет подменять реализацию БД без изменения сервисов.
- Dependency Injection (DI / IoC): С помощью @RequiredArgsConstructor и final полей внедрены зависимости через конструктор. Это реализация принципа инверсии управления.
- Specification (Паттерн «Спецификация»): Класс HotelSpecifications — это реализация паттерна для построения гибких бизнес-правил поиска.
- Embedded Value (Value Object): Объекты Contacts и ArrivalTime помечены как @Embeddable. Это паттерн для группировки связанных полей в одну логическую единицу внутри таблицы.
